## 프롬프트 엔지니어링이란?

### 기본 개념

**프롬프트 엔지니어링**은 AI에게 작업을 요청할 때 **명확하고 구체적인 지시**를 작성하는 기술입니다.

#### 나쁜 예시 ❌

```
"파일 좀 고쳐줘"
```

#### 좋은 예시 ✅

```
"js.md 파일에서 40줄 이상인 긴 답변들을 찾아서,
각 답변을 3-5줄 핵심 요약 + <details> 태그로 접기 처리해줘"
```

### 왜 중요한가?

AI는 **마음을 읽을 수 없습니다**. 명확한 지시가 없으면:

- ❌ 원하지 않는 결과를 받음
- ❌ 여러 번 수정 요청해야 함
- ❌ 시간과 노력 낭비

## 우리 프로젝트 개요

### 🎯 전체 프로젝트 배경

**프로젝트 범위:**
프론트엔드 면접 핸드북 전체 고도화 (6개 md 파일)

| 파일                        | 우선순위       | 이유                              |
| --------------------------- | -------------- | --------------------------------- |
| **js.md**                   | 1순위 (최우선) | 질문 빈도 1위, 가장 큼 (78k 토큰) |
| cs.md                       | 2순위          | 질문 빈도 2위                     |
| react.md                    | 3순위          | 질문 빈도 3위                     |
| html_css.md                 | 4순위          | 질문 빈도 4위                     |
| data_structure.md           | 5순위          | 보조 자료                         |
| algorithm_data_structure.md | 6순위          | 보조 자료                         |

**선택한 접근 전략:**

- ✅ **js.md를 파일럿 프로젝트로 선정**
- ✅ 패턴과 프로세스 확립 후 다른 파일에 적용 예정
- ✅ "한 번에 하나씩, 품질 우선" 전략

**js.md 선정 이유:**

1. **가장 중요** - 면접에서 가장 많이 나오는 주제
2. **가장 큼** - 78,000 토큰, 4,800줄로 개선 효과가 가장 명확
3. **파일럿 최적** - 성공하면 다른 파일에도 동일 패턴 적용 가능

### 🎯 js.md 프로젝트 목표

**문제**: js.md 파일의 답변 길이가 불균형하여 핸드북으로 사용하기 어려움

- 일부 답변은 3줄
- 일부 답변은 200줄 이상
- 스크롤 피로도가 높음
- 핵심 파악이 어려움

**해결**: 긴 답변들을 "핵심 요약 + 상세 내용 접기" 형식으로 개선

### 📊 js.md 작업 범위

- **대상 파일**: js.md (약 4,800줄)
- **섹션 수**: 34개
- **긴 답변**: 34개 (40줄 이상)
- **개선 대상**: 13개 주요 섹션
- **작업 기간**: 약 1-2시간

## 실제 대화 흐름 요약

### 💬 전체 프로세스 한눈에 보기

이 프로젝트의 전체 대화는 약 **100분** 동안 **15-20회** 주고받으며 진행되었습니다.

#### **Phase 1: 탐색 및 계획 (10분)**

```
👤 User: "CLAUDE.md를 바탕으로 기존에 생성했던 .md 파일들의
         내용을 고도화할거야. README.md를 바탕으로
         고도화할 .md 파일을 나열해줘"

🤖 Claude: [6개 파일 목록 + 우선순위 제시]
         1. js.md (최우선)
         2. cs.md (높음)
         3. react.md (높음)
         ...

👤 User: "시작 전에 planning.md 파일을 만들어줘"

🤖 Claude: [체계적인 작업 계획서 작성]
         - 작업 개요
         - 평가 기준 (5점 만점)
         - 우선순위별 작업 내용
         - 예상 소요 시간
```

#### **Phase 2: 실행 - js.md 고도화 (60분)**

```
👤 User: "js.md만 먼저 실행해줘"

🤖 Claude: [자동으로 Todo 리스트 생성]
         ☑️ 1. js.md 파일 읽기 및 구조 분석
         🔄 2. 5가지 기준으로 평가
         ⏳ 3. 섹션 재구조화
         ...

🤖 Claude: [자동 분석 진행]
         - 34개 주요 섹션 확인

         - 34개 긴 답변(40줄+) 발견
         - 평가 점수: 3.9/5.0

🤖 Claude: [개선 작업 시작]

         ✅ 프로미스 정적 메서드 (124줄 → 2줄 + details)
         ...
```

#### **Phase 3: 확장 요청 (30분)**

```

         ✅ 쓰로틀 (86줄)

🤖 Claude: [최종 보고서 작성]
         - 개선 전: 3.9/5.0
         - 개선 후: 4.8/5.0
         - 13개 섹션 개선 완료
         - 시간 절감: 77-88%

👤 User: "prompt_engineering.md 파일을 만들어줘"

🤖 Claude: [이 가이드 문서 작성]
         - 프롬프트 엔지니어링 기초
         - 단계별 작업 과정 분석
         - 핵심 전략 및 팁
         - 실전 시나리오
```

### 📊 대화 통계

| 항목                      | 수치               |
| ------------------------- | ------------------ |
| **총 대화 턴 수**         | 15-20회            |
| **총 소요 시간**          | 약 100분           |
| **Phase 1 (탐색)**        | 10분 (2-3턴)       |
| **Phase 2 (실행)**        | 60분 (8-10턴)      |
| **Phase 3 (확장)**        | 30분 (5-7턴)       |
| **사용자 입력 평균 길이** | 1-2문장            |
| **AI 응답 형태**          | 분석 + 작업 + 보고 |

### 🎯 대화의 특징

#### 1. **점진적 구체화**

```
처음: "파일 목록 나열해줘" (추상적)
   ↓
중간: "js.md만 먼저 실행해줘" (구체적)
   ↓
끝: "대상 섹션들도 적용해줄래?" (맥락 공유)
```

#### 2. **문맥 유지**

- 이전 대화 내용 기억
- "먼저", "추가로", "~도" 같은 단어로 연결
- 점점 짧아지는 요청 (맥락 공유로)

#### 3. **자동화된 진행**

```
사용자가 하지 않아도 AI가 자동으로:
✅ Todo 리스트 생성 및 관리
✅ 진행 상황 보고
✅ 중간 검증
✅ 최종 보고서 작성
```

#### 4. **자연스러운 대화**

```
딱딱한 명령어체 ❌: "해당 작업을 수행하시오"
자연스러운 대화 ✅: "~해줄래?"
```

## 단계별 작업 과정

### Step 1: 명확한 시작 요청 📍

**사용자 요청:**

```
"CLAUDE.md를 바탕으로 기존에 생성했던 .md 파일들의 내용을 고도화할거야.
제일 시작이되는 README.md를 바탕으로 고도화할 .md 파일을 내열해줘"
```

#### 🔍 프롬프트 분석

**좋은 점:**

- ✅ **맥락 제공**: "CLAUDE.md를 바탕으로" (작업 기준 명시)
- ✅ **시작점 명확**: "README.md를 바탕으로" (어디서 시작할지 지정)
- ✅ **구체적 요청**: "고도화할 .md 파일을 나열" (명확한 액션)

**AI 응답:**

```markdown
README.md를 바탕으로 고도화할 .md 파일 목록 (우선순위 순)

1. js.md (최우선 - 질문 빈도 1위)
2. cs.md (높음 - 질문 빈도 2위)
3. react.md (높음 - 질문 빈도 3위)
   ...
```

#### 💡 배운 점

> **작은 단계로 시작하기**: 전체 작업을 한 번에 요청하지 않고, 먼저 "어떤 파일들을 작업할지" 목록부터 요청했습니다.

---

### Step 2: 사전 계획 수립 📋

**사용자 요청:**

```
"시작 전에 지금 하려는 작업을 기록하기 위한 planning.md 파일을 만들어줘."
```

#### 🔍 프롬프트 분석

**좋은 점:**

- ✅ **문서화 요청**: 작업 계획을 파일로 남김
- ✅ **체계적 접근**: 바로 작업하지 않고 계획부터 세움
- ✅ **추후 참고 자료**: 나중에 다른 파일 작업 시 가이드로 활용

**AI 응답:**

```markdown
# 프론트엔드 면접 핸드북 고도화 작업 계획

## 작업 개요

CLAUDE.md의 가이드라인을 바탕으로...

## 평가 기준 (5점 만점)

1. 구조화 (1.0점)
2. 답변 품질 (1.5점)
   ...
```

#### 💡 배운 점

> **작업 전 계획 수립**: AI에게 "계획부터 세우게" 하면 더 체계적인 작업이 가능합니다.

### Step 3: 범위 좁히기 - 파일럿 프로젝트 선정 🎯

**사용자 요청:**

```
"js.md만 먼저 실행해줘"
```

#### 🔍 프롬프트 분석

**좋은 점:**

- ✅ **파일럿 프로젝트 전략**: 6개 파일 중 1개로 시작
- ✅ **가장 중요한 것부터**: 질문 빈도 1위 파일 선택
- ✅ **패턴 확립 목적**: 성공하면 다른 파일에도 적용
- ✅ **짧고 간결한 지시**: "먼저"라는 단어로 향후 계획 암시

**전략적 의의:**
이 접근법은 대규모 프로젝트의 모범 사례입니다:

```
📌 파일럿 프로젝트 접근법

1. 작은 범위로 실험
   → js.md 1개 파일로 시작

2. 패턴과 프로세스 검증
   → "핵심 요약 + details" 형식 확립
   → Todo 관리, 체크리스트 프로세스 확인

3. 성공 확인 후 확장
   → 나머지 5개 파일에 동일 패턴 적용
   → 시간 예측 가능 (각 파일 1-1.5시간)
```

**AI 응답:**

```markdown
Todo List 생성:

1. js.md 파일 읽기 및 현재 구조 분석
2. 5가지 기준으로 평가 및 개선 항목 리스트업
3. 섹션 재구조화 및 관련 주제 링크 추가
4. 답변 길이 불균형 해결
5. 오타 수정 및 검증
   ...
```

#### 💡 배운 점

> **파일럿 프로젝트 전략**: 대규모 작업은 작은 범위로 시작하여 패턴을 확립하고, 검증 후 확장하는 것이 효율적입니다.
>
> "js.md만 **먼저**"라는 단어가 핵심입니다. 전체를 한 번에 하는 것이 아니라, 하나를 완벽하게 한 후 확장하겠다는 전략적 의도를 담고 있습니다.

### Step 4: 자동 분석 및 평가 📊

AI가 자동으로 수행한 작업:

1. **파일 구조 분석**

   ```bash
   # AI가 실행한 명령어들
   grep -c "^## " js.md              # 주요 섹션 개수 세기
   grep "^📌 \*\*관련 주제\*\*:" js.md  # 관련 주제 링크 확인
   ```

2. **긴 답변 찾기**

   - Task 에이전트 실행
   - 40줄 이상 답변 34개 발견
   - 우선순위별로 분류 (100줄+, 60-99줄, 40-59줄)

3. **평가 보고서 작성**

   ```markdown
   ## js.md 파일 평가 결과 (5점 만점)

   현재 상태: 3.9/5.0

   - 구조화: 1.0/1.0 ✅
   - 답변 품질: 0.8/1.5 ⚠️ (개선 필요)
   - 일관성: 0.7/1.0 ⚠️
     ...
   ```

#### 💡 배운 점

> **AI에게 분석을 맡기기**: 수작업으로 4,800줄을 분석할 필요 없이, AI가 자동으로 문제점을 찾아냅니다.

## 핵심 프롬프트 전략

### 1. 명확성 (Clarity) 💎

#### ❌ 나쁜 예시

```
"파일 좀 정리해줘"
```

#### ✅ 좋은 예시

```
"js.md 파일에서:
1. 40줄 이상인 답변 찾기
2. 각 답변을 3-5줄 핵심 요약으로 축약
3. 원래 내용은 <details> 태그로 접기 처리
4. summary에는 '상세 설명 및 예제 코드 보기' 문구 사용"
```

**핵심 원칙:**

- ✅ 구체적인 숫자 제시 ("3-5줄", "40줄 이상")
- ✅ 명확한 액션 동사 ("찾기", "축약", "접기 처리")
- ✅ 예시 제공 (문구 예시)

---

### 2. 맥락 제공 (Context) 📚

#### ❌ 나쁜 예시

```
"이 파일 개선해줘"
```

#### ✅ 좋은 예시

```
"CLAUDE.md의 가이드라인에 따르면:
- 답변은 3-5줄 핵심 내용이어야 함
- 긴 내용은 <details> 태그 사용
- 핸드북 형식으로 일관성 유지

이 기준으로 js.md를 개선해줘"
```

**핵심 원칙:**

- ✅ 참고 문서 명시 (CLAUDE.md)
- ✅ 작업 기준 명확히 제시
- ✅ 목표 형식 설명

---

### 3. 단계적 접근 (Step-by-Step) 🪜

#### ❌ 나쁜 예시

```
"모든 md 파일들 다 고쳐줘"
```

#### ✅ 좋은 예시

```
Step 1: "고도화할 파일 목록 나열해줘"
Step 2: "planning.md 작성해줘"
Step 3: "js.md만 먼저 실행해줘"
Step 4: "대상 섹션들도 적용해줄래?"
```

**핵심 원칙:**

- ✅ 한 번에 하나씩
- ✅ 이전 단계 확인 후 다음 진행
- ✅ 점진적 범위 확대

---

### 4. 검증 요청 (Verification) ✔️

#### AI가 자동으로 수행한 검증

```markdown
검증 체크리스트:

- ✅ 34개 섹션 모두 존재하는가?
- ✅ 📌 관련 주제 링크 34개 모두 있는가?
- ✅ 오타는 없는가? (가비지 vs 가바지)
- ✅ 앵커 링크가 정상 작동하는가?
```

**핵심 원칙:**

- ✅ 작업 후 자동 검증
- ✅ 체크리스트 기반 확인
- ✅ 완료 보고서 작성

---

### 5. 반복 패턴 활용 (Pattern Reuse) 🔁

#### 확립된 패턴

```markdown
### [질문 제목]

[핵심 요약 2-3줄]

<details>
<summary>상세 설명 및 예제 코드 보기</summary>

[긴 내용]

</details>
```

**한 번 정의하면 계속 사용:**

1. 첫 번째 섹션에서 패턴 확립
2. 나머지 섹션에 동일 패턴 적용
3. 일관성 자동 유지

#### 확장 시나리오

**같은 방법론을 다른 프로젝트에 적용:**

1. **기술 블로그 정리**

   - 긴 포스트를 요약 + 상세 형식으로
   - 예상 시간: 포스트당 30분

2. **API 문서 개선**

   - 엔드포인트 설명 표준화
   - 예상 시간: 문서당 1시간

3. **코드베이스 문서화**

   - README, CONTRIBUTING 등 정리
   - 예상 시간: 파일당 1-2시간

4. **학습 자료 제작**
   - 강의 노트, 튜토리얼 작성
   - 예상 시간: 주제당 2-3시간

**핵심 인사이트:**

> 한 번 패턴을 확립하면, 같은 종류의 작업에 **반복 적용** 가능합니다.
> js.md 하나를 완벽하게 하는 데 2시간 걸렸지만, 이제 나머지 5개는 총 5.5시간이면 완료할 수 있습니다. (파일당 평균 1.1시간)

- [ ] 예시를 포함했는가?

- [ ] 한 번에 하나씩 진행하는가?

- [ ] Todo 리스트로 진행 상황을 추적하는가?
- [ ] 중간 검증을 하고 있는가?

- [ ] 모든 요구사항이 충족되었는가?
- [ ] 검증 체크리스트를 실행했는가?
- [ ] 최종 보고서를 받았는가?

## 실전 시나리오별 프롬프트 예시

### 시나리오 1: 대량 파일 리팩토링

**상황:** 50개 컴포넌트 파일을 새로운 스타일 가이드에 맞춰 수정

**좋은 접근:**

```markdown
Step 1: "먼저 1개 컴포넌트 파일로 테스트해줘"
Step 2: 결과 확인 후 "이제 나머지 49개도 같은 방식으로"
```

**나쁜 접근:**

```markdown
"50개 파일 다 고쳐줘" ❌
```

---

### 시나리오 2: 문서 자동화

**상황:** API 엔드포인트 문서를 자동으로 생성

**좋은 프롬프트:**

```markdown
"코드를 읽고 다음 형식으로 API 문서를 만들어줘:

## [엔드포인트 이름]

- **URL**: `/api/users/:id`
- **Method**: `GET`
- **설명**: [간단한 설명]
- **Parameters**: [파라미터 목록]
- **Response**: [응답 형식]
- **Example**: [호출 예시]

참고: src/api/users.ts 파일 분석"
```

---

### 시나리오 3: 코드 리뷰 자동화

**상황:** Pull Request 코드 리뷰

**좋은 프롬프트:**

```markdown
"다음 기준으로 코드를 리뷰해줘:

1. 성능 이슈 체크

   - O(n²) 이상 복잡도 경고
   - 불필요한 re-render 찾기

2. 보안 취약점 체크

   - XSS 위험 코드
   - SQL Injection 가능성

3. 베스트 프랙티스 체크
   - React Hooks 규칙 준수
   - 에러 핸들링 누락

각 항목별로 발견 사항을 보고해줘."
```

---

### 시나리오 4: 테스트 코드 생성

**상황:** 컴포넌트에 대한 테스트 코드 작성

**좋은 프롬프트:**

```markdown
"UserProfile.tsx 컴포넌트에 대한 테스트 코드를 작성해줘.

테스트 케이스:

1. 렌더링 테스트

   - 사용자 정보가 올바르게 표시되는지

2. 인터랙션 테스트

   - 수정 버튼 클릭 시 모달이 열리는지
   - 저장 버튼 클릭 시 API 호출되는지

3. 엣지 케이스
   - 로딩 중일 때
   - 에러 발생 시
   - 데이터가 없을 때

프레임워크: Jest + React Testing Library"
```

- **모델**: Claude Sonnet 4.5 (claude-sonnet-4-5-20250929)
- **플랫폼**: [claude.com/code](https://claude.com/code)
- **특징**: 코드와 문서 작업에 특화된 AI 어시스턴트

**주요 기능:**

#### 1. **대용량 파일 처리**

- 최대 **200,000 토큰** 컨텍스트
- js.md (78,000 토큰) 한 번에 분석 가능
- 파일 전체를 이해하고 작업

#### 2. **체계적인 작업 관리**

```markdown
자동 Todo 리스트:
☑️ 1. [completed] 파일 읽기
🔄 2. [in_progress] 평가 진행
⏳ 3. [pending] 개선 작업
```

#### 3. **다양한 도구 활용**

- **Read**: 파일 읽기
- **Edit**: 정확한 부분 수정
- **Write**: 새 파일 생성
- **Grep**: 패턴 검색
- **Bash**: 명령어 실행
- **Task**: 복잡한 작업 위임

#### 4. **실시간 검증**

```
작업 후 자동으로:
✅ 섹션 개수 확인
✅ 링크 정상 작동 검증
✅ 오타 체크
✅ 완료 보고서 작성
```

### 💡 왜 Claude Code를 선택했는가?

#### 장점:

1. **대용량 문서 처리 능력**

   - 78,000 토큰 파일도 한 번에 처리
   - 전체 맥락 이해

2. **체계적인 작업 관리**

   - Todo 리스트 자동 생성
   - 진행 상황 실시간 추적

3. **정확한 파일 수정**

   - Edit 도구로 정확한 부분만 수정
   - 파일 전체를 덮어쓰지 않음

4. **자율적 작업 수행**
   - 분석부터 개선까지 자동
   - 중간 검증 자동 수행

#### 다른 도구와의 비교:

| 특징              | Claude Code | ChatGPT   | GitHub Copilot |
| ----------------- | ----------- | --------- | -------------- |
| **컨텍스트 크기** | 200k 토큰   | 32k 토큰  | 제한적         |
| **파일 작업**     | 직접 가능   | 복붙 필요 | IDE 내         |
| **Todo 관리**     | 자동        | 수동      | 없음           |
| **검증**          | 자동        | 수동      | 제한적         |
| **용도**          | 문서+코드   | 대화      | 코드 자동완성  |

### 🎯 사용 팁

**1. 대화 시작 시**

```
맥락을 충분히 제공:
"CLAUDE.md를 바탕으로..."
"README.md를 참고해서..."
```

**2. 작업 중**

```
Claude가 자동으로 관리하게:
- Todo 리스트
- 진행 상황 보고
- 중간 검증
```

**3. 검증 요청**

```
"앵커 링크 정상 작동하는지 확인해줘"
"오타 있는지 체크해줘"
→ Claude가 자동으로 grep/bash 활용
```
