# CLAUDE.md

이 파일은 Claude Code (claude.ai/code)가 이 레포지토리에서 작업할 때 참고할 가이드입니다.

## 레포지토리 개요

이 레포지토리는 개발자들이 기술 면접을 준비하도록 돕기 위해 설계된 한국어 프론트엔드 면접 준비 핸드북입니다. JavaScript, React, Computer Science 기초, HTML/CSS, 자료구조 및 알고리즘을 다루는 종합적인 학습 자료를 포함하고 있습니다.

콘텐츠는 일반적인 면접 순서가 아닌 관련 개념별로 순차적으로 구성된 참고 핸드북 형식으로 구조화되어 있습니다. 불 이모지 표시(🔥)는 자주 묻거나 중요한 질문을 나타냅니다.

## 레포지토리 구조

### 주요 콘텐츠 파일

- **js.md** - JavaScript 기초 (변수, 데이터 타입, 클로저, async/await, 프로미스 등)

- **html_css.md** - HTML/CSS 기초 (시멘틱 태그, 웹 표준, 접근성, CSS 포지셔닝, flexbox/grid)
- **data_structure.md** - 자료구조 (리스트, 큐, 스택, 해시 테이블, 트리, 힙, 그래프)
- **algorithm_data_structure.md** - 알고리즘 패턴과 구현 (상세 구현이 포함된 v2)

### 보조 파일

- **question_list.md** - 실제 기업 면접에서 받은 질문들 (A-F 기업) 및 자가 평가 (✅ 답변함 / ❎ 어려웠음)

### 디렉토리

  - **useState.jsx** - useState 훅 예제
  - **useReducer.tsx** - useReducer 훅 예제

- **.github/ISSUE_TEMPLATE/** - 기여를 위한 이슈 템플릿

## 개발 환경

### 필수 도구

- **Node.js**: v22 이상
- **pnpm**: 패키지 매니저

### 설치 및 실행

```bash
# 의존성 설치
pnpm install

# VitePress 로컬 개발 서버 (http://localhost:5173)
pnpm run docs:dev

# 프로덕션 빌드 (docs/ 동기화 포함)
pnpm run docs:build

# 빌드 결과물 미리보기
pnpm run docs:preview

# 루트 마크다운 → docs/ 수동 동기화
pnpm run sync:docs
```

### 배포 구조

루트 마크다운 파일이 단일 소스(Single Source of Truth)이며, 다음 흐름으로 배포됩니다:

```
루트 .md 파일 → (sync-docs.js) → docs/ → (VitePress) → Vercel 자동 배포
```

- **pre-commit hook** (husky): 커밋 시 `sync:docs` 자동 실행 → `docs/*.md` 자동 스테이징
- **Vercel**: push/PR마다 `docs:build` 실행, 빌드 실패 시 배포 차단

### 마크다운 포맷 및 린트

```bash
# Prettier로 마크다운 포맷 통일
pnpm run format

# 포맷 검사만 (수정 없이)
pnpm run format:check

# 전체 콘텐츠 파일 린트
pnpm run lint:md

# 자동 수정 가능한 항목 수정
pnpm run lint:md:fix

# 내부 앵커 링크 검증 (외부 URL 제외)
pnpm run lint:links
```

### 관련 주제 링크 패턴

각 주요 섹션(## 레벨)에는 **📌 관련 주제** 링크를 추가하여:

- 섹션 간 연결성을 명확히 표시
- 학습자가 관련 개념을 쉽게 탐색할 수 있도록 지원
- 각 섹션마다 2-4개의 관련 주제 링크를 포함 (너무 많지 않게)

형식:


## 기여 가이드라인

콘텐츠를 추가하거나 수정할 때는 다음 단계를 따르세요:

1. **콘텐츠 추가** - 적절한 `.md` 파일에 질문과 답변 작성
2. **목차 업데이트** - 새로운 섹션/질문을 다음에 추가:
   - 해당 마크다운 파일의 내부 목차
   - README.md 메인 목차
3. **마크다운 앵커 형식 사용** - 질문은 다음 패턴 사용: `[제목 🔥](#앵커-링크)`
4. **이슈 생성** - `.github/ISSUE_TEMPLATE/-issues-xxx--xx---.md`의 이슈 템플릿 사용
   - 체크리스트 항목: ✓ 내부 목차에 추가함, ✓ README 목차에 추가함
5. **Pull Request 제출** - PR에서 이슈 참조

## 답변 형식 가이드라인

답변은 다음과 같아야 합니다:

- **간결함** - 핵심 개념에 집중한 3-5줄
- **키워드 중심** - 면접관이 기대하는 기술 용어 강조
- **실용적** - 관련 있는 경우 실제 사용 맥락 포함
- **점진적** - 단순하게 시작하고, 후속 질문에 대해 더 깊은 설명 제공

### 답변 길이 불균형 해결

답변이 40줄 이상으로 길어지는 경우:

1. **핵심 답변 추출** - 3-5줄로 핵심 내용만 요약
2. **상세 설명 접기** - `<details>` 태그로 추가 설명, 예제 코드, 이미지 등을 접기 처리
3. **일관성 유지** - 모든 답변이 유사한 길이로 통일되도록 관리

예시:

```markdown
### 질문 제목

핵심 답변을 3-5줄로 작성합니다. 가장 중요한 개념과 키워드를 포함합니다.

<details>
<summary>상세 설명 및 예제 보기</summary>

여기에 코드 예제, 이미지, 자세한 설명을 포함합니다.

</details>
```

### 평가 기준 (5점 만점)

1. **구조화** (1.0점)
   - 논리적인 섹션 구분이 있는가?
   - 관련 주제 링크(📌)가 적절히 배치되어 있는가?
   - 목차가 잘 정리되어 있는가?

2. **답변 품질** (1.5점)
   - 답변이 3-5줄의 핵심 내용을 포함하는가?
   - 기술 용어와 키워드가 적절히 사용되는가?
   - 긴 답변(40줄 이상)은 `<details>` 태그로 접기 처리되었는가?

3. **일관성** (1.0점)
   - 답변 길이가 균형있는가?
   - 형식이 통일되어 있는가?
   - 불 이모지 사용이 일관적인가?

4. **정확성** (1.0점)
   - 오타나 문법 오류가 없는가?
   - 기술적 내용이 정확한가?
   - 링크가 올바르게 작동하는가?

5. **완성도** (0.5점)
   - 누락된 섹션이 없는가?
   - 예제 코드가 적절히 포함되어 있는가?
   - 이미지/다이어그램이 필요한 곳에 배치되어 있는가?

## 주의사항 (Gotchas)

- **docs/ 디렉토리를 직접 수정하지 마세요** — `docs/*.md`는 `sync-docs.js`에 의해 루트 파일에서 자동 생성됩니다. 항상 루트의 `.md` 파일을 수정하세요.
- **이미지 경로는 루트 기준으로 작성하세요** — 루트 파일에서는 `./images/`, `./cs_images/`를 사용합니다. `sync-docs.js`가 VitePress용 절대 경로(`/images/`)로 자동 변환합니다.
- **이미지 확장자 대소문자에 주의하세요** — `.JPG`, `.PNG` 등 대문자 확장자는 `sync-docs.js`가 소문자로 자동 변환하지만, 실제 파일명과 일치하는지 확인이 필요합니다.
- **앵커 링크는 반드시 소문자로 작성하세요** — 마크다운 헤딩의 앵커 ID는 소문자로 생성됩니다. `(#React-기초)` ✗ → `(#react-기초)` ✓. `lint:links`로 검증할 수 있습니다.
- **앵커 링크의 🔥 이모지 처리** — VitePress는 헤더의 이모지를 앵커 생성 시 제거합니다. 목차에서 앵커 링크를 작성할 때 이모지를 포함하지 않도록 주의하세요.
- **`<details>` 태그 앞뒤로 빈 줄 필수** — 마크다운 파서가 HTML 블록을 올바르게 인식하려면 `<details>` 및 `</details>` 태그 앞뒤에 빈 줄이 있어야 합니다.
- **pre-commit hook이 자동 실행됩니다** — 커밋 시 `format → lint:md → sync:docs → 스테이징` 순서로 실행됩니다. 린트 에러가 있으면 커밋이 차단됩니다.
- **개발 환경/도구 변경 시 ADR을 작성하세요** — 새로운 도구 도입, 아키텍처 변경, 워크플로우 변경 등은 `adr/` 디렉토리에 ADR을 작성하여 의사결정 맥락을 기록합니다. 기존 ADR의 결정이 변경되면 상태를 "대체됨"으로 업데이트하세요.

## 이 레포지토리 작업 시

콘텐츠를 수정할 때:

- Q&A 핸드북 구조 유지
- 한국어를 전체적으로 보존
- 답변을 간결하고 키워드 중심으로 유지 (3-5줄)
- 긴 답변은 `<details>` 태그로 접기 처리
- 주요 섹션에 📌 관련 주제 링크 추가
- 개별 파일의 목차와 README.md 모두 업데이트
- 면접 빈도에 기반하여 질문 중요도를 나타내는 불 이모지 사용
- 마크다운의 앵커 링크가 올바르게 작동하는지 확인
